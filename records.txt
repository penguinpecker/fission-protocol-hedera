═══════════════════════════════════════════════════════════════════════
CHILD-RECORD POLICY  (deterministic 2-tx — NO atomic, NO fallbacks)
═══════════════════════════════════════════════════════════════════════

DECISION (2026-05-27): every Buy/Sell flow is explicitly 2 transactions
from day 1. No atomic-1-tx attempts. No fallback flags. Each tx peaks
at ~25 child records — half the 50 limit. Works regardless of future
Hedera precompile changes or SaucerSwap NPM changes. No version-coupling
to downstream gas costs.

  43. Periphery exposes paired entrypoints only:
        Buy:    zapHbarToSy(deadline)                                    → Tx 1
                buySyForPt(market, syIn, minPtOut, deadline)             → Tx 2
                buySyForYt(market, syIn, minYtOut, deadline)             → Tx 2
                buySyForLp(market, syIn, minLpOut, deadline)             → Tx 2

        Sell:   sellPtForSy(market, ptIn, minSyOut, deadline)            → Tx 1
                sellYtForSy(market, ytIn, minSyOut, deadline)            → Tx 1
                sellLpForSy(market, lpIn, minSyOut, deadline)            → Tx 1
                unzapSyToHbar(syAdapter, sharesIn, minHbarOut, deadline) → Tx 2

  44. NO single-tx zapHbarToPt/Yt/Lp. NO sellPtForHbar single-tx.
      The contract literally does not expose them — design enforces 2-tx.

  45. Frontend submits Tx1, waits for receipt, then submits Tx2 using
      the SY balance delta. Uses the same wallet session — no extra
      connects. UI shows "Step 1/2 → Step 2/2" progress.

  46. Exception: standalone HBAR ↔ SY (zapHbarToSy / unzapSyToHbar)
      is naturally 1-tx — no curve trade, no second leg required.

  47. ZERO testnet child-count gate. Each leg's child budget is
      bounded by the structure (~25 each). No edge cases to engineer
      around. No "the day Hedera bumps an opcode" risk.

FORENSIC AUDIT  (2026-05-27, post-deploy)
═══════════════════════════════════════════════════════════════════════

Scope: SaucerSwapLPYieldSource, FissionRewardsMarket, FissionPeriphery.
Verdict: NO CRITICAL FINDINGS. Architecture sound. Listed limitations are
owner-trust / documentation concerns acceptable for current TVL (operator
seed only); revisit before scaling user TVL.

HIGH (no immediate fix; documented limitations):
  H-1. Periphery.rescueToken — owner can drain Periphery's transient
       balances. Defense-in-depth: every flow refunds dust in-tx. Real
       risk = orphaned tokens from a failed inner call. Mitigation:
       transfer Periphery ownership to Timelock + ThresholdAccount.
  H-2. FissionRewardsMarket._split settles ytReceiver but not ptReceiver
       reward state. Re-analysis: proportional reward distribution
       preserved by pre-mint ts snapshot in _harvestRewards — NOT actually
       exploitable. Audit flag was a misanalysis.
  H-3. Market.swapExactYtForSyFor has no _checkSize gate (Periphery does).
       Direct integrators could brick the pool with one huge YT sell.
       Documented intent: gate lives in Periphery only.
  H-4. Periphery.unzapSyToHbar accepts arbitrary syAdapter. Mitigation
       options: (a) gate on registeredSyAdapter mapping (requires Periphery
       v2 redeploy) or (b) trust the frontend to always pass the
       canonical syAdapter. Currently (b) applies.

MEDIUM (cosmetic / acceptable):
  M-1. Direct user-to-user SY share transfers leak rewards. Acknowledged
       in SYBase; all SY movement is mediated by Market in practice.
  M-2. Constructor doesn't validate rewardTokens ≠ shareToken/PT/LP. Risk
       only if SY ever returns conflicting tokens. SaucerSwapLPYieldSource
       returns (token0, token1) = (USDC, WHBAR) which are not collisions.
  M-3 / M-4 / M-5: cosmetic.

LOW:
  L-1. deadline=0 sentinel allowed (by design).
  L-2. min*Out=0 allowed (user-set slippage).
  L-3. Periphery.buySyForPt returns minPtOut not actual swap output
       (cosmetic; curve guarantees ≥minPtOut).
  L-4. zapHbarToSy forwards address(this).balance (includes stray HBAR).
  L-5. _redeemSyToHbar uses 0 mins; outer min*Out gates.
  L-6. initialize has no initialAnchor sanity bounds (admin-trust).

INFO:
  - First-deposit attack: mitigated via MINIMUM_LIQUIDITY=1000.
  - Re-entrancy: all entries use ReentrancyGuardTransient.
  - Pause: redeemLiquidity, merge, redeemAfterExpiry are intentionally
    NOT pause-gated (users must always be able to exit).
  - HTS Ed25519 long-zero: handled via _ytBal mirror.
  - Curve init: setInitialLnImpliedRate validates newExchangeRate ≥ 1.

CONCLUSION: shippable. Address H-4 (registered-adapter gate) when next
contract cycle happens. Transfer ownership to Timelock before scaling TVL.

X-7 + X-11 RESOLUTION  (2026-05-27, post-cascade)
═══════════════════════════════════════════════════════════════════════

X-7 (market setOperator permanent) — NOT a contract bug, design choice.
  - The audit-noted concern was: combined with Periphery's `rescueToken`
    powers, the operator-approval trust surface extends to "future owners
    of the Periphery".
  - Periphery v3 closed the rescueToken concern via isProtectedToken
    (USDC/WHBAR/PT/LP/SY-share are unrescuable). So even a malicious
    future Periphery owner can't drain protocol tokens.
  - Operator approval is freely revocable: market.setOperator(periphery,
    false). Demonstrated by revoking v1 + v2 on old market (txs
    0xe92210... + 0x7793fd...).
  - FRONTEND UX TODO: surface approved-operators in user profile / settings
    with revoke buttons. Contract behavior is correct.

X-11 (old Periphery v1 immutable) — risk reanalyzed and largely closed.
  - The concern was: v1's standing user approvals + no `registeredSyAdapter`
    gate means a user could be tricked into calling v1.unzapSyToHbar with
    a malicious syAdapter, getting their SY siphoned.
  - REANALYSIS: v1.unzapSyToHbar does `safeTransferFrom(msg.sender, v1,
    sharesIn)` — pulls from msg.sender, NOT an arbitrary owner. Operator's
    standing approval to v1 cannot be leveraged by an attacker because
    v1 never does `transferFrom(operator, ...)` with an attacker as caller.
  - The ACTUAL risk vector: a victim cached on v1 in their browser calls
    v1.unzapSyToHbar(maliciousAdapter, ...) themselves. They'd lose their
    own SY. Mitigation = frontend cache invalidation (Vercel rebuild
    handles this on next visit).
  - DEFENSIVE ACTIONS TAKEN:
      ✓ setOperator(v1, false) on old market — closes operator-callable
        path. Tx 0xe92210089e...
      ✓ setOperator(v2, false) on old market — same. Tx 0x7793fdd2...
      ✗ ERC-20 approve(spender, 0) on HTS facade reverts (known Hedera
        quirk; the facade doesn't accept zero-amount approves cleanly).
        Standing PT/LP/SY-share allowances remain non-zero but are NOT
        exploitable by external callers per the reanalysis above.
